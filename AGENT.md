# 「浮宠」V1.0 MVP — 产品需求文档（PRD v2）

## 一、文档基础信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 浮宠 |
| 版本号 | V1.0 安卓 MVP 版 |
| 迭代周期 | 4 周（开发 3 周 + 测试 1 周） |
| 核心目标 | 跑通「获取宠物素材 → 桌面悬浮交互 → 渐进式免打扰」核心闭环；次日留存 ≥40%，7 日留存 ≥20% |
| 版本边界 | 仅安卓端；零服务端（App 纯客户端）；不上无障碍服务；无内置预设宠物库；MVP 仅 2D，架构预留 3D 扩展点 |

| 架构原则 | **纯客户端运行**，无自建后端；用户自主配置 AI 服务（自带 API Key）；宠物素材全部本地存储；权限按需渐进获取，非必要不申请 |

---

## 二、产品整体架构

MVP 仅 2 个载体：

1. **桌面悬浮层（核心使用场景）**：桌宠常驻载体，负责渲染、触摸交互、自主动作、自动避让逻辑
2. **APP 主界面（素材获取 + 设置入口）**：负责宠物素材导入/生成、AI 服务配置、权限管理、免打扰设置；用户无需频繁打开

**与 v1 的关键差异：**
- 无服务端，所有网络请求仅发生于用户配置的 AI 服务端点
- 用户照片仅发送至用户指定的 AI 服务，App 不做中转、不留存上传记录
- AI 生成为可选增强功能，非强制路径——用户可直接导入本地图片作为桌宠素材
- **渲染层抽象**：动画采用逐帧精灵图方案（非骨骼绑定），悬浮窗通过 `PetRenderer` 接口隔离具体渲染实现；MVP 用 Canvas + `FrameSequencer`，后续换 3D 渲染器（Filament / SceneViewer）时悬浮窗、交互、自主动作模块无需重写

## 三、核心功能详细需求

### 模块 1：宠物素材获取（P0）

#### 1.1 功能描述
用户通过**本地导入**或**配置 AI 服务后生成**两种方式获取桌宠素材。动画方案采用**逐帧精灵图**（Sprite Sheet）：每个动作由 1-N 张独立 PNG 帧组成，播放时按帧序列切换。不使用统一骨骼变形——同一只宠物的每个动作帧均由 AI 独立生成，确保每只宠物的每个动作都是独特的。

#### 1.2 素材来源

**路径 A：本地导入（默认路径，零门槛）**
- 支持从相册选取图片或直接拍照
- 导入后进入「素材编辑页」，提供：
  - 裁剪（自由比例或固定 1:1）
  - 简易抠图（纯色背景或自动识别主体边缘，手动橡皮擦辅助）
  - 双指缩放调整默认显示大小
  - 拖拽调整素材锚点（宠物"站立"的基准位置）
- **单张静态图**：导入后宠物仅具备待机动画——App 对图片施加微弱的呼吸式缩放+轻微位移动画（非骨骼变形，仅 Canvas 变换），使静态图有「活着」的感觉。所有触摸交互仍可触发，但宠物反馈为图片整体位移/缩放/晃动，无逐帧动作变化
- **GIF / 多张序列帧**：App 自动拆帧，用户进入「动作分组编辑页」，将帧拖入不同动作组（idle / greet / sleep / sit / stretch / shrink / happy / sad），每组 2-5 帧。分组完成后即可投放桌面，各动作按对应帧序列播放。若某动作未分配帧，则该交互触发时播放 idle 帧 + 整体变换作为回退
- 导入完成即视为宠物创建成功，可直接投放桌面

**路径 B：AI 逐帧生成（可选增强路径）**
- 用户需先配置 AI 服务方可使用此路径
- 入口位于素材编辑页底部：「AI 生成专属形象」，未配置时按钮置灰 + 提示「配置 AI 服务后可用」
- 配置流程：
  1. 用户进入「设置 → AI 服务配置」
  2. 选择服务商（预置选项：OpenAI 兼容接口 / 通义千问 / 火山引擎 等通用协议端点）
  3. 填入 API Key 及自定义 Endpoint（如需）
  4. App 调 `/models`（或等效接口）验证连通性，返回「配置成功」或错误信息
  5. API Key 本地加密存储（AES + Android Keystore），不上传任何第三方
- **渐进式生成流程**（核心帧优先 + 后台补全）：
  1. 用户上传 3-9 张宠物照片
  2. App 本地压缩图片（JPEG 80%，最长边 ≤1024px）
  3. **第一阶段——核心帧生成**（预计 1-2 分钟）：AI 生成 3 组核心动作帧：
     - idle（待机，3-5 帧）：宠物正常站姿、轻微呼吸起伏
     - greet（迎接，2-3 帧）：抬头、身体微前倾
     - sleep（睡觉，2-3 帧）：蜷缩或趴下闭眼
  4. 核心帧生成完成 → 系统通知 + 展示预览，用户可立即「放到桌面」，宠物已具备基础交互能力
  5. **第二阶段——补充动作后台生成**（不阻塞用户使用）：其余 6 组动作（sit / stretch / shrink / happy / sad / walk）在后台逐个生成，每完成一组自动补充到宠物素材库，宠物能力逐步丰富
  6. 每个动作的生成请求独立发送，单帧失败仅重试该帧（最多重试 2 次），不影响已生成的其他帧
  7. 所有帧生成完成后系统通知：「你的宠物已解锁全部互动动作」
  8. 生成超时（单动作默认 90 秒）→ 该动作标记为「未生成」，对应交互使用 idle 帧 + 整体变换回退
  9. 生成失败 → 展示明确的 HTTP 错误信息（如 401/429/500），引导用户检查 API Key 或余额

**AI 角色一致性策略：**
- 每帧生成 prompt 均包含同一份「角色描述」（从用户照片提取的关键视觉特征：品种、毛色、花纹、体型、耳朵形状等）
- 角色描述由首次调用时通过一次额外的 vision API 提取，后续所有帧复用该描述
- 用户可在素材编辑页手动修正角色描述（如"白色比熊犬，圆脸，棕色鼻子"），修正后全部帧可重新生成

**3D 版本预留：** AI 配置页、请求构造、结果处理均预留 `generation_type` 参数（MVP 固定 `2D_IMAGE`）。3D 版本新增 `3D_MODEL` 生成类型，请求/响应格式走独立分支（`model/gltf+json`），超时独立配置。详见第四章。

#### 1.3 素材格式与动画

**MVP（2D 逐帧精灵图）：**

素材存储结构：
```
{internal_storage}/pet_data/{pet_id}/
  ├─ pet.json              # 元数据（动作→帧映射、锚点、大小等）
  ├─ idle/
  │   ├─ frame_01.png      # 透明背景 PNG
  │   ├─ frame_02.png
  │   └─ frame_03.png
  ├─ greet/
  │   ├─ frame_01.png
  │   └─ frame_02.png
  ├─ sleep/
  │   └─ ...
  ├─ sit/
  ├─ stretch/
  ├─ shrink/
  ├─ happy/
  ├─ sad/
  └─ walk/
```

**动画规则：**
- 每个动作目录下 1-5 帧，帧数不足 2 张时该动作播放单帧 + Canvas 整体变换（位移/缩放/旋转）作为回退
- 待机（idle）为默认循环动作，始终在播放；其他动作为一次性/临时动作，播放完毕后回到 idle
- 帧切换间隔可配置（默认 150-300ms/帧），在 `pet.json` 中按动作独立设置
- 帧序列支持正向/循环/乒乓（正向播完反向回）三种播放模式
- 所有帧均为带透明通道的 PNG，确保悬浮窗无背景遮挡桌面
- 素材不暴露给系统相册（存储于 App 内部目录）

**本地导入的三种素材形态：**

| 导入形态 | 动画能力 | 说明 |
| --- | --- | --- |
| 单张静态图 | 仅 idle（呼吸式缩放+微位移） | 无逐帧变化，所有交互以整体变换回退 |
| GIF / APNG | 自动拆帧 → 用户分组到动作 | 帧数取决于原文件 |
| 多张图片批量导入 | 用户在编辑页手动拖入各动作组 | 最灵活，可混搭来源 |

**3D 版本预留：**
- 素材管线支持扩展 3D 模型格式：`glTF 2.0`（`.gltf`/`.glb`）作为标准交换格式
- 3D 模型文件含：网格 + 骨骼（Skeleton）+ 蒙皮数据 + 动画片段（Animation Clips）
- 素材存储目录按素材类型分子目录（`2d/`、`3d/`），MVP 仅使用 `2d/`

### 模块 2：桌面悬浮交互体系（P0）

#### 2.1 功能描述
宠物常驻手机桌面（以悬浮窗形式），支持触摸交互，根据手机状态触发自主动作，行为逻辑符合真实宠物特征；无任何广告弹窗打扰。

#### 2.2 悬浮窗基础规则
- 使用 `WindowManager` + `TYPE_APPLICATION_OVERLAY` 实现
- 宠物默认常驻所有桌面页面，不随桌面翻页移动，固定在用户设置的位置
- 支持长按拖拽到屏幕任意位置，松手时自动吸附屏幕边缘（边缘留 8dp 安全间距）
- 层级：低于系统状态栏、导航栏、弹窗、键盘；高于桌面图标和应用内容

**MVP 渲染方案（2D 逐帧精灵图）：**
- 渲染引擎：基于 Android Canvas 自绘，不引入第三方动画库
- 核心为 `FrameSequencer`：根据当前动作从对应目录按序取帧 Bitmap，按配置间隔切换
- 帧播放支持正向循环（idle/walk）、正向单次（greet/shrink）、乒乓循环（stretch）
- 每帧绘制前可选叠加 Canvas 变换（整体缩放/位移/旋转），用于单帧回退和呼吸效果
- 帧率目标：60fps；息屏/锁屏时停止渲染

**3D 版本预留——渲染层抽象：**
- 悬浮窗持有 `PetRenderer` 接口，不直接依赖具体渲染引擎：
  ```
  interface PetRenderer {
      fun init(surface: Surface, width: Int, height: Int)
      fun loadPet(assetPath: String, type: PetAssetType)  // type: 2D_SPRITE | 3D_MODEL
      fun playAction(action: PetAction)                   // idle/walk/sleep/…
      fun render(frameTimeNanos: Long)
      fun hitTest(x: Float, y: Float): PetBodyPart?       // 2D 按比例区域命中 / 3D 射线命中
      fun applyTransform(scale: Float, dx: Float, dy: Float, rotation: Float)  // 整体变换（单帧回退用）
      fun dispose()
  }
  ```
- MVP 提供 `SpriteRenderer`（2D 逐帧实现，内部含 `FrameSequencer`）；后续新增 `GLTFRenderer`（3D 实现，基于 Filament / OpenGL ES / Vulkan），上层悬浮窗、触摸交互、自主动作模块**零改动**
- 3D 渲染器要求：独立 `SurfaceView` 或 `TextureView` 嵌入悬浮窗 Window，可在 2D 和 3D 之间切换；共享同一套 `PetAction` 动画指令集

#### 2.3 触摸交互（响应延迟 ≤100ms）

| 交互动作 | 宠物反馈 |
| --- | --- |
| 单指点击宠物 | 抬头 + 眨眼 + 轻微身体晃动 |
| 单指双击宠物 | 蹭手动作 + 短促音效（需音效开启） |
| 单指在宠物上滑动（摸头/身体） | 身体跟随手指方向微移，做眯眼/放松姿态 |
| 单指戳宠物腹部 | 身体收缩 + 后退一小步 |
| 双指缩放 | 实时调整宠物大小，松手后自动保存 |
| 长按 2 秒 | 进入可拖拽状态，周围浮现半透明删除/隐藏按钮；拖至删除按钮临时隐藏，可在 App 内恢复 |

音效默认关闭，用户可在设置中开启。

**触摸命中检测（不依赖骨骼）：**
- MVP 2D：命中检测基于宠物精灵图的整体包围盒 + 按比例区域划分
  - 精灵图纵向三等分：上 1/3 = 头部区域、中 1/3 = 身体区域、下 1/3 = 腹部区域
  - 命中后返回 `PetBodyPart` 枚举（HEAD / BODY / BELLY），上层按区域分发到对应动作
  - 若该动作对应的帧序列未生成/未分配，回退为整体变换（缩放/位移）响应
- 3D 版本：命中检测走 `PetRenderer.hitTest(x, y)` → 射线与骨骼蒙皮网格求交。上层触摸分发逻辑完全不感知底层是 2D 还是 3D——`hitTest` 返回同一个 `PetBodyPart` 枚举，交互反馈逻辑不改一行代码

#### 2.4 自主动作（默认低频率）

宠物根据**设备状态 / 环境 / 用户场景**三类信号自主触发动作。所有自主动作 1 小时内最多触发 3 次（可在设置调整频率或完全关闭）。整体遵循权限渐进原则：权限越多感知越丰富，零权限也能用核心悬浮交互。

##### 2.4.1 设备状态触发（零权限或系统广播）

| 触发场景 | 所需权限 | 宠物动作 |
| --- | --- | --- |
| 用户解锁手机 | 无（`ACTION_USER_PRESENT` 广播） | 抬头 + 打招呼 |
| 手机接入充电 | `ACTION_POWER_CONNECTED` 广播 | 移动到屏幕下方充电口一侧，趴下休息 |
| 电量低于 20% | `ACTION_BATTERY_LOW` 广播 | 耷拉脑袋，蔫蔫状 |
| 桌面无操作超 5 分钟 | 悬浮窗内部计时 | 舔毛 / 伸懒腰 / 趴下睡觉 |

##### 2.4.2 环境感知触发（仅网络，零权限）

天气数据通过免费天气 API 获取（如和风天气，每天调用 3-4 次足够），不需要精确定位，使用粗略城市级位置即可；时间段基于系统时钟，零成本。

| 触发场景 | 数据来源 | 宠物动作 |
| --- | --- | --- |
| 晴天 | 天气 API | 开心晃头，做出晒太阳的放松姿态 |
| 雨天 | 天气 API | 蜷缩成一团，做出"甩水/躲雨"动作 |
| 雪天 | 天气 API | 兴奋跳跃 |
| 高温（>32℃） | 天气 API | 趴着吐舌头喘气 |
| 低温（<5℃） | 天气 API | 蜷缩发抖 |
| 清晨（6:00-9:00） | 系统时钟 | 伸懒腰、打哈欠（刚睡醒） |
| 深夜（23:00 后） | 系统时钟 | 打哈欠，趴下，做出"困了"的动作 |

天气感知可在设置中关闭；关闭后不发起任何网络请求。

##### 2.4.3 用户场景感知触发（需 Usage Access 权限，可选）

基于前台 APP 分类，宠物做出陪伴式反应（不打扰，仅情绪化表现）。包名→分类映射表见**附录 A**。

| 检测到用户在使用 | 宠物行为 |
| --- | --- |
| 社交类（微信/QQ/微博/小红书） | 安静趴着，偶尔偷偷瞄一眼屏幕 |
| 游戏类（王者/原神/吃鸡） | 躲到角落，变小 + 降低透明度，不打扰 |
| 视频类（B站/腾讯/爱奇艺/抖音） | 坐到屏幕下方一角，面朝屏幕"一起看" |
| 音乐类（网易云/QQ音乐/Spotify） | 闭眼随节奏微微晃动，像在听音乐 |
| 办公类（钉钉/飞书/腾讯会议/WPS） | 完全静默，不触发任何自主动作 |
| 阅读类（微信读书/知乎/今日头条） | 趴在屏幕边缘安静陪伴，偶尔翻身 |
| 购物类（淘宝/京东/拼多多） | 正常待机，偶尔好奇歪头 |
| 切回桌面 / 无前台 APP | 轻微晃动身体迎接，恢复自由活动 |

**说明：** 用户场景感知与模块 3 的 APP 黑白名单共用同一份 Usage Access 权限和前台检测能力。若该 APP 同时在黑名单中，"隐藏"优先级高于"陪伴反应"——直接隐藏，不做动作。

##### 2.4.4 权限渐进总览

| 授权情况 | 宠物能感知到 |
| --- | --- |
| 零权限（仅悬浮窗+存储） | 解锁/充电/电量/无操作/时间段 |
| + 网络（默认有） | 以上 + 天气 |
| + Usage Access（可选） | 以上 + 用户使用场景（APP 分类陪伴） |

### 模块 3：渐进式免打扰体系（P0）

#### 3.1 设计理念
不再追求"全场景智能感知"，改为**权限驱动 + 用户手动配置**的分层免打扰：基础层零权限可用，增强层用户主动授权后启用。每项免打扰能力均对应一个 Android 权限，用户可逐项决策。

#### 3.2 权限分层

| 能力 | 所需权限 | 层级 |
| --- | --- | --- |
| 悬浮窗基础显示 | `SYSTEM_ALERT_WINDOW` | 必须（核心功能） |
| 素材本地存储 | `READ/WRITE_EXTERNAL_STORAGE` | 必须（核心功能） |
| 手动免打扰时段（定时静默） | 无 | 基础层（默认可用） |
| 天气感知动作 | 仅网络（`INTERNET`，普通权限免授权） | 基础层（默认可用） |
| 通话中自动隐藏 | `READ_PHONE_STATE` | 增强层（可选授权） |
| 指定 APP 打开时自动隐藏 + 用户场景陪伴反应 | `PACKAGE_USAGE_STATS`（Usage Access） | 增强层（可选授权） |

**仅 2 个强制权限**：悬浮窗 + 存储（与 v1 目标一致）。通话检测和 Usage Access 是可选增强，不授予不影响核心体验。Usage Access 一次授权同时驱动「APP 黑白名单隐藏」和「用户场景陪伴反应（2.4.3）」两项能力。

#### 3.3 手动免打扰设置
- **免打扰时段**：支持添加多个时段（如工作日 9:00-18:00、睡眠 23:00-7:00），时段内宠物缩小为屏幕边缘半透明静默状态，不做自主动作
- **APP 黑白名单**（需 Usage Access 权限）：
  - 黑名单模式（默认）：添加游戏/视频/办公类 APP，打开时宠物自动隐藏，退出后恢复
  - 白名单模式：仅在添加的 APP 内显示宠物
  - 提供快速添加入口：一键勾选抖音、王者荣耀、微信、腾讯会议、哔哩哔哩等高频 APP
- **一键隐藏/显示**：通知栏快捷开关，无需进 App

#### 3.4 自动避让（需对应权限，非强制）

| 触发场景 | 所需权限 | 自动行为 |
| --- | --- | --- |
| 用户接打电话 | `READ_PHONE_STATE` | 宠物自动完全隐藏，通话结束后恢复 |
| 用户打开黑名单 APP | `PACKAGE_USAGE_STATS` | 宠物自动完全隐藏，退出后恢复 |

**明确做不到的（本版本不做，也不作为验收项）：**
- 键盘弹出时自动避让（桌面层级无法感知键盘状态）
- 全屏视频/游戏自动缩小（桌面层级无法感知全屏状态）
- 系统相机打开时自动隐藏（桌面层级无法感知相机状态）

这些场景由用户通过手动免打扰时段 + APP 黑名单间接覆盖。全场景自动感知将在后续版本评估是否引入无障碍服务后实现。

#### 3.5 权限弹窗与授权流程

权限按「用到才申请」原则触发，全程不强制、不连续骚扰。用户实际遇到的弹窗/跳转如下：

```
首次打开 APP
    │
    ├─ 创建宠物 → 导入照片
    │     └─ [系统原生弹窗] 存储/相册权限   ← 拒绝则无法导入，可重试
    │
    └─ 宠物生成完成 → 点击「放到桌面」
          └─ [App 引导页] 说明悬浮窗权限用途
                └─ 点击「去开启」→ [跳转系统设置页] 手动打开开关
                      └─ 若拒绝 → [App 二次提示] "没有悬浮窗权限宠物无法显示，
                                   可后续在设置开启"，提供「暂不开启」，不强制

可选增强权限（仅在用户主动开启对应功能时触发，不主动弹）：
    │
    ├─ 开启「通话时自动隐藏」开关
    │     └─ [系统原生弹窗] 通话状态权限
    │
    └─ 开启「APP 场景感知 / 黑白名单」开关
          └─ [App 引导页] 说明 Usage Access 用途
                └─ 点击「去开启」→ [跳转系统设置页] 在列表中找到本 APP 打开开关
```

**弹窗规则：**
- 系统原生权限弹窗（存储、通话状态）：调用系统 API 触发，文案由系统控制
- 悬浮窗、Usage Access：Android 不提供运行时弹窗，只能跳转系统设置页，故每个配「App 引导页」解释用途后再跳转
- 每次打开 APP 若未开启悬浮窗权限，首页顶部展示一条轻量提示条（非弹窗），不阻断操作
- 所有可选权限拒绝后，对应功能优雅降级，核心悬浮交互不受影响
- **MVP 不使用任何系统级打扰弹窗**：无开屏弹窗、无升级弹窗、无评分弹窗

#### 3.6 宠物气泡对话（MVP 不做，V1.1 预留）

宠物气泡（头顶冒出文字框，如"今天好热啊"）技术上由悬浮窗内部 Canvas 渲染，不经过系统通知、不算系统弹窗。MVP 阶段**不实现气泡内容**，原因：① 文案/多语言维护成本；② 气泡频率控制不当易变骚扰，与免打扰定位冲突。MVP 所有交互反馈通过**动作 + 可选音效**表达。

**但气泡是 V1.1 的必做交互**（动作只能表达情绪，气泡才能传递具体信息），因此 MVP 必须预留完整架构口子：`BubbleLayer` 接口 + 行为引擎统一输出 `PetResponse{action, bubble?}` + 两段式渲染顺序。详见 [4.5 气泡沟通预留](#45-气泡沟通预留v11-必做交互)。MVP 用空实现（`NoOpBubbleLayer`）占位，V1.1 接入气泡时不改行为引擎与触发逻辑。

## 四、扩展性预留设计（V1.1+ 基础设施）

以下内容**不纳入 MVP 开发范围**，但 MVP 阶段必须按此接口约定编码，确保 V1.1 引入 3D 渲染、气泡沟通时无需重构核心模块。两大预留方向：**3D 渲染**（4.1-4.4）与**气泡沟通**（4.5）。

### 4.1 核心抽象层

```
┌─────────────────────────────────────────────┐
│  悬浮窗 (OverlayWindow)                      │
│  ├─ 位置/层级/拖拽/缩放          ← 不变       │
│  ├─ 触摸分发                      ← 不变       │
│  ├─ PetRenderer (接口)           ← 多态切换    │
│  │    ├─ SpriteRenderer (MVP)    ← 2D 实现    │
│  │    └─ GLTFRenderer (V1.1+)   ← 3D 实现    │
│  └─ BubbleLayer (接口)           ← 气泡预留    │
│       ├─ NoOpBubbleLayer (MVP)   ← 空实现     │
│       └─ TextBubbleLayer (V1.1+) ← 气泡渲染   │
├─────────────────────────────────────────────┤
│  行为引擎 (PetBehaviorEngine)                 │
│  └─ 输出 PetResponse{action, bubble?}        │
│     ├─ action  → PetRenderer.playAction()    │
│     └─ bubble? → BubbleLayer.show()  (MVP 忽略)│
├─────────────────────────────────────────────┤
│  素材管线 (AssetPipeline)                     │
│  ├─ 导入 → 处理 → 本地存储                    │
│  └─ PetAssetType: 2D_SPRITE | 3D_MODEL       │
└─────────────────────────────────────────────┘
```

**不变量（贯穿 2D/3D、有无气泡）：**
- `PetAction` 枚举：`IDLE | WALK | SLEEP | SIT | STRETCH | GREET | SHRINK | HAPPY | SAD` ——两套渲染器共用同一套动作语义
- `PetBodyPart` 枚举：`HEAD | BODY | BELLY`（2D 按精灵图纵向 1/3 分区；3D 按骨骼蒙皮命中）——触摸命中区域语义统一
- `PetResponse` 数据类：`{ action: PetAction, bubble: BubbleContent? }` ——**行为引擎统一输出模型**。MVP 阶段 `bubble` 恒为 `null`，所有触发场景只填 `action`；V1.1 引入气泡时只需让行为规则填充 `bubble` 字段，行为引擎与触发逻辑零改动
- `PetState` 数据类：位置、大小、透明度、当前动作、是否静默 ——与渲染维度无关
- 悬浮窗 Window 管理：`WindowManager` 参数、层级、吸附逻辑、生命周期 ——完全不变

**可变点（通过 `PetRenderer` 接口隔离）：**
- 初始化方式：2D 用 Bitmap/Canvas + `FrameSequencer`；3D 用 Surface + OpenGL 上下文
- 资源加载：2D 加载 PNG 帧序列并按动作分组；3D 加载 glTF/GLB 含骨骼动画
- 渲染循环：2D 用 `invalidate()` + `onDraw()`；3D 用独立的 GL 渲染线程（`GLThread`）
- 命中检测：2D 用包围盒 + 纵向比例分区（上/中/下 1/3）；3D 用射线投射

### 4.2 3D 素材支持规划

| 项目 | MVP（2D） | V1.1+（3D） |
| --- | --- | --- |
| 素材格式 | PNG 帧序列（按动作分目录）/ GIF | glTF 2.0（.gltf / .glb） |
| 动画数据 | 逐帧精灵图（每动作 1-5 帧） | 骨骼动画片段（Animation Clip） |
| 获取方式 | 本地导入 / AI 逐帧生成 | 本地导入 .glb / AI 生成 3D 模型 |
| 渲染引擎 | Canvas 自绘 + FrameSequencer | Filament / SceneViewer / OpenGL ES |
| 碰撞检测 | 包围盒 + 纵向比例分区（1/3 段） | 射线-三角形求交 |
| 内存占用 | ≤80MB（含帧序列位图，按动作懒加载） | ≤120MB（含模型+纹理，V1.1 重新核定） |

### 4.3 AI 生成 3D 模型预留

- AI 服务配置页面的「生成类型」参数预留 `2D_IMAGE | 3D_MODEL` 枚举，MVP 固定为 `2D_IMAGE`
- 3D 生成 API 协议与 2D 共用同一套认证（用户 API Key），仅 `response_format` 不同（`image/png` vs `model/gltf+json`）
- 3D 模型生成耗时显著长于 2D（分钟级 vs 秒级），需独立超时配置（默认 300 秒），进度回调接口预留 `progress: Float` 字段

### 4.4 触摸交互 3D 扩展

- 3D 宠物支持「旋转查看」：单指在宠物上横向滑动 → 宠物绕 Y 轴旋转，纵向滑动 → 绕 X 轴轻微俯仰。与 2D「摸头/身体」的滑动反馈走不同事件通道
- 3D 专属手势「双击旋转」：双击后进入自由旋转模式，再次双击退出，期间单指滑动 = 旋转，双指缩放 = 拉近拉远相机
- 这些手势仅 3D 渲染器激活时生效，2D 模式下滑动手势行为不变

### 4.5 气泡沟通预留（V1.1 必做交互）

气泡是宠物**情感表达的核心升级路径**——动作只能表达情绪，气泡能表达具体信息（"今天好热"/"你玩了 2 小时了"/"主人早上好"）。MVP 不实现气泡内容与文案，但**必须把架构口子留好**，使 V1.1 接入气泡时无需改动行为引擎与触发逻辑。

**MVP 阶段必须落地的预留点：**

1. **`BubbleLayer` 接口**：悬浮窗除 `PetRenderer` 外，再持有一个 `BubbleLayer`：
   ```
   interface BubbleLayer {
       fun show(content: BubbleContent)   // 展示气泡
       fun dismiss()                      // 立即收起
       fun isShowing(): Boolean
   }
   data class BubbleContent(
       val text: String,                  // 气泡文字
       val style: BubbleStyle,            // 普通 / 提醒 / 情绪
       val durationMs: Long,              // 自动消失时长
       val anchor: BubbleAnchor           // 相对宠物的方位（上/左/右）
   )
   ```
   - MVP 提供 `NoOpBubbleLayer`：所有方法空实现，不渲染任何东西、不占内存
   - V1.1 提供 `TextBubbleLayer`：Canvas 在悬浮窗内绘制气泡框 + 文字（不经过系统通知，不算系统弹窗），支持淡入淡出、自动消失、点击关闭

2. **行为引擎统一输出 `PetResponse`**（见 4.1 不变量）：所有触发场景（设备状态 / 天气 / 时间 / APP 分类）的行为规则，输出结构统一为 `{ action, bubble? }`。MVP 所有规则的 `bubble` 字段恒为 `null`，由 `NoOpBubbleLayer` 消费，无副作用。

3. **悬浮窗渲染顺序预留**：渲染管线固定为「先渲染宠物（PetRenderer）→ 再渲染气泡（BubbleLayer）」的两段式，气泡层级永远在宠物之上。MVP 气泡层为空，但渲染调用链已存在，V1.1 无需调整渲染顺序。

**MVP 不做的（留 V1.1）：** 具体气泡文案、多语言、触发频率策略、气泡样式设计、用户开关。这些都是「填空」，不动架构。

**气泡频率与免打扰的关系（V1.1 设计约束，提前记录）：** 气泡必须受免打扰体系约束——免打扰时段内、黑名单 APP 内、办公场景下一律不弹气泡。这条规则在 MVP 的行为引擎里已经天然成立（这些场景宠物本就静默），V1.1 只需让气泡复用同一套静默判断。

## 五、全局强制规则

### 5.1 性能验收标准（一票否决）

| 指标 | 标准 |
| --- | --- |
| 后台内存占用 | ≤80MB（含悬浮窗进程），3D 版本 V1.1 重新核定上限（预估 ≤120MB） |
| 整机功耗 | 亮屏每小时 ≤1%，待机 ≤0.5%；3D 版本需独立评估 GPU 功耗 |
| 渲染帧率 | 宠物动作稳定 60fps，无肉眼可见卡顿 |
| 稳定性 | 连续运行 24 小时无崩溃、无 ANR |
| 进程规则 | 无后台自启动、无关联启动；仅用户主动打开 App 或悬浮窗运行时激活进程 |
| 保活策略 | 不自启、不互相唤醒；悬浮窗进程使用前台服务（`startForeground`）+ 常驻通知，通知内容：「桌面宠正在运行」，通知可被用户隐藏（通知渠道设为 LOW 优先级） |

### 5.2 隐私与合规

- **用户照片**：仅在用户主动选择「AI 生成」且配置了 AI 服务时，发送至用户指定的第三方 API 端点；App 自身不做上传、中转、留存。导入素材全部本地存储。
- **API Key**：使用 Android Keystore 加密存储，永不离开设备，不包含在任何日志或崩溃报告中。
- **申请权限**：
  - 强制权限（2 项）：悬浮窗、存储
  - 可选权限（2 项）：通话状态（通话隐藏）、Usage Access（APP 黑白名单）
  - 所有可选权限在对应功能首次使用时单独弹窗申请，附带说明用途，拒绝不影响核心功能
- **无广告**：MVP 无开屏广告、无弹窗广告、无悬浮广告
- **网络请求**：仅发生于用户配置 AI 服务后的生成请求，无后台数据上报、无埋点 SDK、无第三方统计

### 5.3 新手引导
- 宠物第一次出现在桌面时，展示半透明提示：「长按可拖拽移动，双击可以互动哦～」，3 秒后消失，仅展示 1 次
- App 内设置页提供「使用帮助」入口，覆盖所有交互与设置的图文说明
